# Bitrix24 → Nextcloud File Uploader

Автоматическая загрузка файлов из элементов смарт-процессов Bitrix24 в Nextcloud с созданием публичных ссылок и записью их обратно в CRM. После успешной загрузки файлы удаляются из Bitrix24.

## Как это работает

1. Bitrix24 отправляет вебхук при создании/обновлении элемента смарт-процесса
2. Для каждого поля из `FIELD_MAP` скрипт смотрит, сколько файлов в нём лежит:
   - **1 файл** — загружается прямо в папку элемента, создаётся ссылка на файл
   - **2+ файлов** — создаётся подпапка с названием поля, все файлы грузятся туда, создаётся ссылка на папку
3. Ссылка записывается в соответствующее поле-ссылку элемента CRM
4. Файлы удаляются из поля Bitrix24

### Структура папок в Nextcloud

```
/Bitrix24/
└── <Название элемента>/
    ├── документ.pdf              ← одиночный файл (ссылка на файл)
    └── <Название поля>/          ← несколько файлов (ссылка на папку)
        ├── файл1.jpg
        └── файл2.jpg
```

## Структура проекта

```
├── src/
│   ├── webhook.php              # Обработчик вебхука Bitrix24 (точка входа)
│   ├── nextcloud_upload.php     # Библиотека для работы с Nextcloud (WebDAV + OCS)
│   ├── crest.php                # Bitrix24 REST API клиент (CRest v1.36)
│   └── settings.php             # Конфигурация (учётные данные, маппинг полей)
├── upload_to_nextcloud.php      # CLI-скрипт для ручной загрузки
└── README.md
```

## Требования

- PHP 8.0+
- PHP `curl` extension
- HTTP-сервер (Apache/Nginx) для публикации `webhook.php`

## Установка

1. Скопировать файлы на сервер
2. Настроить `src/settings.php`:

```php
define('C_REST_WEB_HOOK_URL', 'https://ваш-битрикс24/rest/1/xxxxxxxxxxxx/');

// поле с файлом => поле для ссылки Nextcloud
define('FIELD_MAP', [
    'UF_CRM_8_XXXXXXX' => 'UF_CRM_8_YYYYYYY',
    'UF_CRM_8_AAAAAAA' => 'UF_CRM_8_BBBBBBB',
]);

define('C_REST_APP_TOKEN', 'ваш-токен-приложения');

define('NC_BASE_URL', 'https://ваш-nextcloud.example.com');
define('NC_USER', 'admin');
define('NC_PASSWORD', 'пароль');
define('NC_REMOTE_DIRECTORY', '/Bitrix24');
```

3. В Bitrix24 создать **исходящий вебхук** на URL `https://ваш-сервер/path/to/src/webhook.php`
   - События: `ONCRMDYNAMICITEMADD`, `ONCRMDYNAMICITEMUPDATE`
   - Указать тот же токен в поле «Ключ» и в `C_REST_APP_TOKEN`

## Настройка полей (`FIELD_MAP`)

```
'UF_CRM_8_<id файлового поля>' => 'UF_CRM_8_<id строкового поля для ссылки>'
```

- **Ключ** — ID кастомного поля типа «Файл» в смарт-процессе
- **Значение** — ID кастомного поля типа «Строка», куда будет записана публичная ссылка

Поля без файлов и поля, у которых ссылка уже заполнена, пропускаются.

## Ручная загрузка

```bash
php upload_to_nextcloud.php
```

Параметры (локальный файл, путь в Nextcloud, опции шаринга) задаются в начале скрипта.

## Логирование

- Вебхук: `src/logs/webhook_YYYY-MM-DD.log`
- CRest: `src/logs/YYYY-MM-DD/HH/`

## Технические детали

- **Потоковая передача файлов** — скачивание из Bitrix24 и загрузка в Nextcloud идут через файловые дескрипторы (`CURLOPT_FILE`, `CURLOPT_INFILE`), без буферизации в RAM. Подходит для больших файлов.
- **Защита от дублей** — при двойном вебхуке (Bitrix24 дублирует событие после `crm.item.update`) второй запрос блокируется файловым локком.
- **Имена папок** — используются человекочитаемые заголовки полей из `crm.item.fields`, кириллица сохраняется.
- **Таймауты** — `set_time_limit(0)` и `ignore_user_abort(true)` для обработки больших пакетов файлов.

## Безопасность

- Вебхук проверяет `C_REST_APP_TOKEN` — запросы без токена возвращают 403
- Авторизация Nextcloud — Basic Auth (учётные данные только в `settings.php`)
- Пароли и куки скрываются из логов (`sanitizeHeadersForLog`)
- **`settings.php` не должен попадать в git** — добавьте его в `.gitignore`
